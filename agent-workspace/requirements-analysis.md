# 患者端需求分析

更新时间：2026-06-26

## 需求来源

本版需求分析基于以下更新后的原始需求重新整理：

- `agent-workspace/interaction-flow.md`：患者端完整业务流程，包括新出诊/复诊入口、检验子流程、单一总超时、处置决策、急症守护、主动退出和 Agent 决策主循环。
- `agent-workspace/core-interaction-flow.md`：核心界面流转，包括阻塞卡、卡片否决、检验缴费、确诊卡、AI 处置决策、完成后继续输入触发自动复诊。
- 当前代码结构：`src/App.tsx` 仍是 Vite 示例页，尚未实现业务界面；后续实现应从状态机、mock facade、聊天时间线和流程卡片骨架开始。

旧版 `interact-flow.md` 已不再作为需求来源；文档引用统一切换为 `interaction-flow.md` 和 `core-interaction-flow.md`。

## 产品定位

患者端是“AI 对话 + 诊疗流程状态机 + 阻塞卡片 + 医疗安全打断”的流程型应用。它不是普通聊天工具，也不是单纯的问答页面。

系统从身份核验和就诊入口开始，区分新出诊与复诊，读取不同上下文；随后通过 Agent 决策主循环持续收集症状、判断是否需要追问或检验，在信息足够时给出诊断并进入处置。全流程由急症守护、整次导诊超时和主动退出结算三个横切机制覆盖。

## 参与方

- 患者：身份核验、描述症状、回答追问、决定是否检验、缴费、执行检验/取药/治疗、查看诊断与医嘱、主动退出、随访中发起复诊。
- AI 导诊 Agent：汇总病史、对话和检验结果；判断信息是否足够确诊；决定追问、检验或确诊；确诊后生成处置方案；持续接收急症守护结果。
- 急症守护 Agent：独立并行监听每轮对话、体征、检验结果和随访状态变化，命中红旗信号时立即打断并提示前往急诊。
- 医院业务系统：身份、病历、历史诊断、检验、支付、药品、治疗、随访、结算和转诊能力。
- 前端系统：将流程状态、卡片动作、阻塞/解锁、终止原因和恢复入口清晰呈现给患者。

## 核心业务目标

1. 支持患者从新出诊或复诊入口进入一次 AI 辅助诊疗流程。
2. 用 Agent 决策主循环表达“继续追问 / 下发检验 / 确诊”的反复推理过程。
3. 用阻塞卡片承载患者必须确认的动作，例如是否检验、检验缴费、药品缴费、取药方式。
4. 将检验定义为纯执行子流程：缴费、采样/执行、结果生成、结果交回 Agent。
5. 处置由 AI 决策，不由患者选择分支；患者只执行分支内动作。
6. 急症、总超时、主动退出在任意状态下拥有高于主流程的优先级。
7. 就诊完成后仍允许输入；继续输入触发复诊评估，而不是追加到已完成会话的普通聊天。

## 范围

### 本期范围

- 身份核验后的患者端问诊入口。
- 新出诊与复诊上下文读取：新出诊读取病历病史；复诊读取病历病史和上次完整诊断记录。
- AI 问诊与 Agent 决策主循环。
- 追问轮次上限与检验轮次上限；达到上限后提示前往大医院。
- 是否检验阻塞卡：同意检验、不查、暂不决定。
- 检验缴费阻塞卡：支付、支付中、支付成功/失败、暂不缴费。
- 检验执行子流程状态：已下发、待缴费、执行中、结果生成、结果回填。
- 确诊卡：诊断结论、依据、风险提示。
- AI 处置决策：用药、治疗、仅医嘱、转大医院。
- 处置执行：药品缴费与取药/配送、自动化治疗预约与执行、仅医嘱展示。
- 随访/复诊提醒，以及病情变化后发起复诊。
- 急症守护、整次导诊超时、主动退出结算三类全局机制。
- 首页历史记录中的继续就诊、复诊、回看入口。

### 暂不纳入

- 医生端、药师端、检验科后台、设备控制台。
- 真实支付、检验设备、药房库存、配送和治疗设备接口实现。
- 医疗知识库、大模型诊断策略、急症规则库本身。
- 后端服务实现和真实鉴权实现；前端需先定义 REST/SSE API contract，并基于 contract 完成 mock。
- 法务合规文本、知情同意和处方审核完整闭环。

## 主流程需求

### 1. 入口与上下文加载

患者打开应用后先完成身份核验，再进入新出诊或复诊入口。

新出诊：

- 读取患者基础信息、病历病史、既往史、过敏史等可用资料。
- 创建新的就诊会话。
- 首条症状输入进入 AI 问诊。

复诊：

- 读取病历病史。
- 读取上次完整诊断记录，包括诊断、处置、检验结果、医嘱和随访信息。
- 进入复诊评估，不应丢失上次上下文。

前端需求：

- 首页支持新建问诊、历史继续、已完成会话复诊。
- 工作台加载时展示上下文加载状态。
- 会话摘要区需要标识“新出诊 / 复诊”。
- 复诊会话必须能引用上次诊断记录，至少在摘要或时间线中可见。

### 2. AI 问诊与 Agent 决策主循环

Agent 每轮基于当前信息做同一套判断：

1. 汇总病史、对话、已有检验结果。
2. 分析病情和可能病症范围。
3. 判断信息是否足够确诊。
4. 信息不足时，选择追问或下发检验。
5. 信息足够时，给出诊断并进入处置决策。

缺症状细节：

- 未达到追问上限时，继续向患者追问。
- 达到追问上限时，提示前往大医院就诊。

缺客观依据：

- 未达到检验上限时，下发检验项，并明确鉴别目标。
- 达到检验上限时，提示前往大医院就诊。

前端需求：

- 支持普通聊天输入和 AI 异步/流式回复。
- 支持展示“正在分析 / 正在生成追问 / 正在判断是否需要检验”等状态。
- 需要在状态机中记录追问轮次和检验轮次。
- 达到任一上限后进入转大医院终止态，不能继续本院流程。
- 决策循环不是线性一次性判断，检验结果回填后必须回到信息汇总再评估。

### 3. 是否检验阻塞卡

当 Agent 判断缺少客观依据且检验轮次未达上限时，下发是否检验卡。

可选动作：

- 同意检验：进入检验缴费卡。
- 不查：明确跳过本轮检验，直接尝试进入诊断评估；如果仍不足以确诊，后续可能继续追问或给出风险提示。
- 暂不决定：否决/挂起当前卡，解锁输入框，允许继续对话；后续 Agent 可再次给出同类卡。

前端需求：

- 卡片展示检验建议原因、检验项目、鉴别目标、费用预估。
- 卡片出现时锁定输入框，并显示锁定原因。
- “不查”和“暂不决定”必须区分：前者是明确跳过检验，后者是继续对话后重新评估。
- 已处理卡片留在时间线中，显示处理结果和时间。

### 4. 检验子流程

检验子流程是执行链路，不再做诊断判断。

流程：

1. Agent 下发检验项。
2. 患者缴纳检验费。
3. 检验科采样或执行。
4. 生成检验结果。
5. 检验结果交回 Agent。
6. Agent 回到决策主循环重新汇总信息。

前端需求：

- 检验缴费卡展示项目、金额、医保/自费信息、支付状态。
- 支持支付中、成功、失败重试。
- 支付成功后展示执行中/等待结果状态。
- 结果回填后插入检验结果摘要，并触发 Agent 重新分析。
- 患者可在缴费、执行、结果等待阶段主动退出；退出后果由不可逆动作决定。

### 5. 确诊卡

当 Agent 判断信息足够时，生成确诊卡。

前端需求：

- 展示诊断结论、诊断依据、风险提示。
- 依据必须区分来自病史、患者回答、检验结果。
- 未做检验时不得伪造检验依据。
- 确诊卡是信息卡，不应要求患者选择诊断分支。
- 确诊后如患者补充新信息，应进入重新评估或复诊机制，不能无状态改写旧结论。

### 6. AI 处置决策

确诊后，由 AI 决策处置方式。

处置类型：

- 用药。
- 自动化治疗。
- 仅医嘱/调养。
- 转大医院。

治疗分支需额外判断本院是否具备能力：

- 具备：安排自动化治疗。
- 不具备：提示前往大医院就诊。

> 后端能力边界：当前 medAgent 决策引擎只产出 `MEDICATION` / `ADVICE_ONLY` / `REFERRAL` 三类处置，**不做院内自动化治疗执行**（需操作/手术直接转诊）。本期"自动化治疗"分支作为产品目标保留，由前端 + mock 先行实现界面与流转，待后端补齐院内治疗执行能力后联调。详见 `special-designs/api.md` 的「与 medAgent 后端的映射与边界」。

前端需求：

- 展示“正在生成处置方案”状态。
- 患者不选择处置分支；正式界面不暴露调试决策按钮。
- 决策为转大医院或本院不具备能力时，进入升级终止。
- 决策结果需要形成可留档的处置方案卡。

### 7. 处置执行

用药：

- 展示药品清单、剂量、用法、禁忌/提醒。
- 支持药品缴费。
- 支持药房取药或配送。
- 完成后进入随访/复诊提醒。

自动化治疗：

- 展示治疗项目、预约/排队/执行状态、注意事项。
- 本院具备能力时才允许安排。
- 完成后进入随访/复诊提醒。

仅医嘱：

- 展示医嘱、观察项、注意事项、复诊建议。
- 完成后进入随访/复诊提醒。

转大医院：

- 展示转诊原因和建议去向。
- 流程升级终止，不再推进本院处置。

### 8. 随访、完成与复诊

处置完成后进入随访/复诊提醒。患者病情未变化且康复达标时，流程结束；若出现新症状或主动点击复诊，回到入口类型判断并按复诊加载上下文。

界面上的就诊完成卡有一个特殊规则：输入框仍可用。患者在完成后继续输入时，系统应触发自动复诊，而不是把消息追加到已完成流程中。

前端需求：

- 完成卡展示本次关键结论、处置摘要、随访/复诊提醒。
- 完成态输入框占位提示“继续描述将自动开始复诊”。
- 完成后新输入触发 `FOLLOW_UP_MESSAGE_SENT` 或等价事件，创建/恢复复诊上下文。
- 历史记录需要区分回看、继续就诊、复诊。

## 全局机制需求

### 急症守护

急症守护独立并行，覆盖问诊、检验、处置、随访全流程。它监听每一次新信息，包括患者输入、体征、检验结果和随访状态变化。

命中急症时：

- 立即打断当前流程。
- 提示患者情况紧急，请立即前往急诊。
- 终止当前线上导诊。

前端需求：

- 急症 overlay 优先级最高。
- 当前 AI 生成、支付等待、检验等待、普通卡片动作都应取消、禁用或标记失效。
- 输入框禁用。
- 终止原因、命中来源、时间需要留档。
- 不实现系统侧急救/转运编排；本期只做“前往急诊”的单一提示。

### 整次导诊单一超时

全流程只有一个总超时计数，作用于一次完整导诊，而不是每个步骤分别计时。

前端需求：

- 会话开始时启动总计时。
- 正常结束时停止计时。
- 超时后进入提示前往大医院的升级终止态。
- 计时状态在顶栏/摘要区可见。
- 超时终止后不得继续推进本院问诊、检验或处置。

### 主动退出与退出结算

患者可在任意步骤主动退出。退出不是普通状态跳转，而是进入统一退出处理。

退出处理按已发生的不可逆动作决定后果：

- 未缴费/未执行：直接结束，无需结算。
- 已缴费未执行：原路退款。
- 已执行检验/治疗：费用照收，结果留档。
- 已取药/已配送：按退药政策处理。

前端需求：

- 顶栏始终提供退出入口。
- 退出前二次确认。
- 确认退出后不可逆。
- 退出结算完成后保存就诊快照。
- 最终状态为就诊结束或挂起，历史记录可回看。

## 状态与事件建模建议

建议使用 XState 建模主流程，并用并行/全局事件处理急症、超时和退出。

主要状态：

- `identityChecking`：身份核验。
- `loadingContext`：加载病史/复诊上下文。
- `chatting`：AI 问诊与追问。
- `analyzing`：Agent 汇总信息并决策下一步。
- `labDecision`：是否检验阻塞卡。
- `labPayment`：检验缴费阻塞卡。
- `labExecution`：检验采样/执行/等待结果。
- `labResultReceived`：结果回填并准备重新分析。
- `diagnosis`：确诊卡。
- `treatmentDecision`：AI 处置决策。
- `medicationPayment`：药品缴费。
- `medicationFulfillment`：取药/配送。
- `treatmentExecution`：自动化治疗预约/执行。
- `adviceOnly`：仅医嘱。
- `followUp`：随访/复诊提醒。
- `completed`：本次就诊完成，输入可触发复诊。
- `emergencyTerminated`：急症终止。
- `transferred`：转大医院/大医院就诊提示。
- `exitSettlement`：退出结算。
- `exited`：退出完成或挂起。

关键事件：

- `IDENTITY_VERIFIED`
- `ENTRY_SELECTED`
- `CONTEXT_LOADED`
- `MESSAGE_SENT`
- `AGENT_ANALYSIS_STARTED`
- `FOLLOW_UP_QUESTION_RAISED`
- `ASK_LIMIT_REACHED`
- `LAB_CARD_RAISED`
- `LAB_ACCEPTED`
- `LAB_SKIPPED`
- `LAB_VETOED`
- `LAB_PAYMENT_SUBMITTED`
- `LAB_PAYMENT_SUCCEEDED`
- `LAB_PAYMENT_FAILED`
- `LAB_RESULT_RECEIVED`
- `LAB_LIMIT_REACHED`
- `DIAGNOSIS_READY`
- `TREATMENT_DECISION_STARTED`
- `TREATMENT_DECIDED`
- `MEDICATION_PAID`
- `MEDICATION_FULFILLED`
- `TREATMENT_COMPLETED`
- `ADVICE_ACKNOWLEDGED`
- `FOLLOW_UP_MESSAGE_SENT`
- `RECOVERY_CONFIRMED`
- `EMERGENCY_DETECTED`
- `VISIT_TIMEOUT`
- `TRANSFER_REQUIRED`
- `EXIT_REQUESTED`
- `EXIT_CONFIRMED`
- `EXIT_SETTLED`

优先级规则：

1. `EMERGENCY_DETECTED` 最高，立即进入急症终止。
2. `EXIT_REQUESTED` 可在任意非终止态触发，但需确认后才结算。
3. `VISIT_TIMEOUT` 高于主流程普通推进，进入转大医院终止。
4. 阻塞卡待处理时，普通 `MESSAGE_SENT` 不应推进，除非患者选择“暂不决定/暂不缴费”解除阻塞。

## 数据对象初稿

### VisitSession

- 会话 ID。
- 患者 ID。
- 入口类型：新出诊、复诊。
- 当前流程状态。
- 开始时间、更新时间、结束时间。
- 总超时截止时间。
- 追问轮次、追问轮次上限。
- 检验轮次、检验轮次上限。
- 上次完整诊断记录 ID（复诊时使用）。
- 终止原因。

### PatientContext

- 患者基础信息。
- 病历病史。
- 既往史。
- 过敏史。
- 当前主诉。
- 上次诊断摘要。
- 上次检验结果摘要。
- 上次处置摘要。

### Message

- 消息 ID。
- 会话 ID。
- 发送方：患者、AI、系统。
- 内容。
- 创建时间。
- 是否流式生成中。
- 关联卡片 ID。
- 是否已被全局打断标记失效。

### FlowCard

- 卡片 ID。
- 会话 ID。
- 卡片类型。
- 卡片状态：待处理、已同意、已跳过、已否决、已支付、已完成、已失效。
- 主要内容。
- 可执行动作。
- 创建时间、处理时间。
- 阻塞原因。

### AgentDecision

- 决策 ID。
- 会话 ID。
- 输入摘要：病史、对话、检验结果。
- 判断结果：需追问、需检验、可确诊、需转大医院。
- 决策理由。
- 追问问题或检验项目。
- 创建时间。

### LabOrder

- 检验单 ID。
- 会话 ID。
- 项目列表。
- 鉴别目标。
- 费用。
- 支付状态。
- 执行状态。
- 检验结果。
- 结果回填时间。

### Diagnosis

- 诊断 ID。
- 会话 ID。
- 诊断结论。
- 诊断依据。
- 依据来源：病史、患者回答、检验结果。
- 是否包含检验依据。
- 风险提示。

### TreatmentPlan

- 处置 ID。
- 会话 ID。
- 类型：用药、治疗、医嘱、转大医院。
- 本院能力判断。
- 执行动作。
- 费用信息。
- 注意事项。
- 完成状态。

### InterruptEvent

- 事件 ID。
- 会话 ID。
- 类型：急症、总超时、主动退出、转大医院。
- 触发来源。
- 触发时间。
- 处理结果。
- 终止原因。

## 关键业务规则

- 新出诊和复诊必须读取不同上下文，复诊不能当作全新无上下文问诊。
- Agent 决策主循环会在追问、检验结果回填后反复运行。
- 追问和检验都必须有轮次上限，达到上限后统一提示前往大医院。
- 检验子流程只负责执行，不在子流程内做诊断分支判断。
- 阻塞卡出现后，主流程不得自动越过该卡。
- “暂不决定/暂不缴费”会解除当前阻塞并恢复对话。
- “不查”不是否决，而是明确跳过检验。
- 检验结果回填后必须交回 Agent 重新汇总分析。
- 处置分支由 AI 决策，患者不直接选择处置类型。
- 本院不具备治疗能力时，进入转大医院终止。
- 急症守护每次新信息都重新筛查，不只在开头筛查一次。
- 整次导诊只有一个总超时计数。
- 主动退出后果由不可逆动作决定，不由当前停在哪个页面决定。
- 完成后继续输入触发自动复诊，不追加到已完成流程。
- 所有终止态都必须记录终止原因。

## 验收标准

- 患者可从新出诊入口开始：身份核验 -> 读取病史 -> 问诊 -> 不查 -> 确诊 -> 处置 -> 随访/完成。
- 患者可从复诊入口开始，并在工作台看到上次完整诊断记录被纳入上下文。
- 患者可走完整检验路径：问诊 -> 是否检验 -> 缴费 -> 执行检验 -> 结果回填 -> Agent 重新分析 -> 确诊。
- 患者在是否检验卡选择暂不决定后，输入框恢复，后续可再次收到检验建议。
- 追问达到上限时，流程进入前往大医院提示。
- 检验达到上限时，流程进入前往大医院提示。
- 检验结果或随访中新出现急症信号时，急症守护立即打断并终止线上导诊。
- 整次导诊超时后，进入转大医院终止态。
- 患者任意状态点击退出并确认后，进入退出结算，并按不可逆动作展示结算结果。
- 处置为本院不具备治疗能力时，进入转大医院终止态。
- 就诊完成后继续输入，触发复诊评估，而不是继续普通聊天。
- 已处理卡片保留在聊天时间线中，并显示处理结果。

## 风险与待确认问题

- 身份核验方式和失败后的前端处理需要确认。
- 新出诊读取的病史字段、复诊读取的“上次完整诊断记录”字段由前端 API contract 先定义，并在 mock 中体现。
- 追问轮次上限、检验轮次上限、总超时时长需要产品和医疗安全规则确认。
- 患者选择“不查”后，如果信息仍不足以确诊，是继续追问、风险提示后给出低置信诊断，还是直接转大医院，需要明确。
- 检验缴费卡“暂不缴费”的业务后果需要明确：取消检验、暂缓检验、继续对话后重新建议，还是保留待支付订单。
- 检验执行中的退出、退款、结果留档规则需要与结算系统对齐。
- 自动化治疗的预约、排队、执行、完成字段尚不清楚。
- 用药分支是否涉及处方审核、药师审核、库存、配送或自取政策，需要补充。
- 急症命中文案、法律责任边界和是否展示命中原因需要单独设计。
- 完成后自动复诊是复用原 session、创建新 session，还是建立父子 session 关系，需要在前端 contract 中先给出默认设计，后续后端按 contract 联调或提出明确变更。
- AI 决策失败、流式中断、支付失败、检验结果延迟、网络断线恢复等异常态需要在前端 API contract 和 mock 中补齐。

## 后续实现建议

1. 先实现基于 XState 的主状态机骨架，覆盖入口、问诊、检验、确诊、处置、完成和三类全局终止。
2. 建立前端 API contract 与 mock facade，模拟新出诊/复诊上下文、Agent 决策循环、检验结果回填和全局打断。
3. 实现工作台聊天时间线与 FlowCard 基础组件，保证卡片混排、阻塞锁定、处理留痕可跑通。
4. 实现首页历史中心，打通新建问诊、继续就诊、复诊、回看。
5. 补齐急症、超时、退出结算 overlay/sheet。
6. 在前端 API contract/schema 明确后细化用药、自动化治疗、结算和复诊会话关系，并在结项时输出 REST API 文档。
