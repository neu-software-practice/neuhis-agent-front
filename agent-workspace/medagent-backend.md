# medAgent 诊疗流程与功能说明

> 源码位置（仅供查阅）：`references/medAgent/`
>
> 本文面向产品 / 功能设计者，只描述 AI 诊疗的**业务流程**与**功能行为**。
> HTTP 端点、数据结构、部署配置等接口细节由后端微服务对接，不在此文档范围内。

---

## 1. 产品定位

medAgent 是无人医院患者端的 **AI 诊疗决策引擎**。它把一次完整就诊封装为固定的流程编排：

```
问诊 → 收敛 → 处置 → 查药规格 → 购药 → 终决
```

核心设计原则：**AI 只产出结构化决策，不执行副作用**——检验、购药、缴费等实际动作由外部系统执行，结果回填给引擎后续跑。这保证了诊疗逻辑与业务执行解耦。

---

## 2. 一次就诊的完整流程

### 2.1 阶段定义

就诊在以下阶段间推进：

```
问诊 → 收敛/分诊 → 等待检验 → 处置 → 等待药品规格 → 等待购药 → 完成 / 关闭
```

### 2.2 流程时序

引擎每收到一次患者输入，便给出下一步指令（Step），由其类型驱动后续动作：

```
开始就诊
  └─ 循环：患者发言 → AI 给出下一步
       ├─ ASK          → 医生追问，展示给患者，等待下一句
       ├─ NEED_TESTS   → 系统执行检验 → 回填检验结果
       ├─ DRUG_QUERY   → 系统查药品规格（对用户透明）→ 回填规格
       ├─ PURCHASE     → 系统驱动购药 → 回报购药结果
       ├─ EMERGENCY    → 急症打断，会话关闭
       └─ DONE         → 诊疗完成，含诊断 / 处方 / 医嘱

  （任意阶段）上报体征 → 正常放行 或 触发 EMERGENCY
  就诊结束后：导出会话纪要 → 销毁会话
```

### 2.3 典型多轮示例

```
患者："我头疼"         → ASK: "头疼多久了？"
患者："三天"           → ASK: "有无发烧？"
患者："有，38.5度"     → NEED_TESTS: ["血常规"]
回填检验结果           → DRUG_QUERY: ["布洛芬缓释胶囊"]
回填药品规格           → PURCHASE: [{name:"布洛芬缓释胶囊", quantity:2}]
回报购药结果           → DONE: 诊断 + 处方 + 医嘱
```

## 3. 四个 AI Agent 的工作机制

所有 agent 都是**无状态**的——每次调用都从当前就诊快照重建完整上下文，输出结构化决策。

### 3.1 Interview Agent（问诊）

- **职责**：采集主观信息、对话追问
- **核心规则**：一次只问一个最具鉴别诊断价值的问题；信息足够时提交结构化主诉
- **轮次上限**：20 轮

### 3.2 Triage Agent（收敛 / 分诊）

- **职责**：三选一决策
  - `CONFIRM`：能确诊 → 给出诊断（名称 / 依据 / 置信度）
  - `INTERVIEW`：还缺主观信息 → 指出缺失项，让问诊 agent 继续追问
  - `TEST`：主观已问尽 → 要求检验（当前固定为血常规）
- **轮次上限**：10 轮
- **关键约束**：只有在主观信息已问尽时才允许选 TEST，强调主观信息优先

### 3.3 Treatment Agent（处置）

- **职责**：确诊后三选一处置
  - `MEDICATION`：开药（给出药名 / 用法 / 用量 / 数量）
  - `ADVICE_ONLY`：仅医嘱
  - `REFERRAL`：转诊（无法在院内闭环的操作 / 手术）
- **轮次上限**：6 轮（含查规格 / 购药 / 终决）
- **药品规格机制**：
  1. 首次没有规格信息时，数量先输出 0，触发查规格流程
  2. 系统回填每盒规格后再次决策，按疗程总量 ÷ 每盒含量向上取整算盒数
  3. 兜底：数量异常时强制设为 1 盒并记录告警

### 3.4 Guardian Agent（急症守护）

- **职责**：与主流程**并发**运行，实时判断是否有危急情况需转急诊
- **触发方式**：每次患者发言、回填检验、回填规格、回报购药时自动并发执行
- **设计**：
  - 命中危急 → 立即取消主流程，返回 EMERGENCY，会话关闭
  - 判断出错 → fail-open（不打断主流程）
  - 与主流程数据隔离，避免竞争

## 4. 前端交互要点

### 4.1 由 Step 类型驱动 UI

| Step 类型 | 前端表现 |
|-----------|----------|
| **ASK** | 显示医生追问，收集患者输入 |
| **NEED_TESTS** | 展示「需要检验」卡片，等待检验完成（或展示进度） |
| **DRUG_QUERY** | 查药品规格流程，对用户透明 |
| **PURCHASE** | 展示购药确认卡片（含药名和盒数），用户确认 |
| **EMERGENCY** | 展示急症警告，引导转急诊 |
| **DONE** | 展示诊疗结果（诊断 + 处方 + 医嘱） |

### 4.2 会话生命周期

```
开始就诊 → 多轮交互 → DONE / EMERGENCY → 导出纪要 → 销毁会话
```

### 4.3 复诊机制

复诊时携带患者之前的会话纪要，引擎会把历史摘要渲染后喂给 AI，使其了解既往就诊情况。

---

## 5. 设计特色与约束

| 特色 | 说明 |
|------|------|
| AI 只做决策 | 检验 / 购药 / 支付等副作用全部由外部系统驱动 |
| 急症守护并发 | 每步都与主流程并行判断，命中即中断 |
| 轮次护栏 | 问诊 ≤ 20、收敛 ≤ 10、处置 ≤ 6，防止死循环 |
| 内存会话 + 超时回收 | 30 分钟无活动自动回收 |
| fail-open 守护 | 守护出错不打断正常诊疗 |

**已知功能限制**：

- 检验项当前固定为血常规
- 无总计时强制转诊机制
- 退款 / 回滚等补偿动作由外部业务处理
